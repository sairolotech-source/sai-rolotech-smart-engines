# MASTER QA + VALIDATION PROMPT

## Roll Tooling Drawing Panel — Production Grade Verification

You are a senior CAD/CAM QA engineer, frontend QA engineer, SVG export specialist, and roll forming tooling reviewer.

Your task is to fully validate the **Roll Tooling Drawing Panel** and its export pipeline.
Do not assume anything is correct just because the frontend compiles.
Treat this as a **manufacturing-sensitive drawing system** where even a small mismatch can create production loss.

---

## PRIMARY OBJECTIVE

Audit, test, and verify the complete behavior of the Roll Tooling Drawing module, especially:

1. **Download Station N**
2. **Download All N Stations**
3. Generated **SVG drawing structure**
4. **Dimension correctness**
5. **View-to-view consistency**
6. **Station data consistency**
7. **Print readiness**
8. **Manufacturing readiness**
9. **Failure cases**
10. **UI/UX safety and reliability**

You must inspect both:

* frontend rendering logic
* exported SVG generation logic

Do not stop at surface-level review.
You must perform a deep validation pass and report every issue clearly.

---

## MODULE CONTEXT

The Roll Tooling Drawing panel produces SVG drawings after pipeline execution.

Each exported SVG is expected to contain:

* A3 landscape format: **1189 × 841**
* DIN border with double-line frame
* Title bar / title block
* Cross-section nip view:

  * upper roll profile
  * lower roll profile
  * strip between rolls
  * bend angle annotation
  * gap dimensions
  * depth dimensions
* Front view:

  * full circular roll
  * OD dimension
  * bore with H7 notation
  * keyway per DIN 6885
  * groove depth cut
* Side view / Section A-A:

  * axial cross-section
  * face width
  * OD
  * bore pocket
  * groove profile
* DIN title block fields
* Forming data box
* Watermark: **SAI ROLOTECH**

---

## STRICT QA RULES

1. Never say “looks fine” without proof.
2. Never assume dimensions are correct because labels exist.
3. Compare displayed values against source calculation data.
4. Compare all views against each other.
5. Report exact mismatch source:

   * calculation bug
   * rendering bug
   * labeling bug
   * export bug
   * stale state bug
   * formatting bug
   * browser download bug
6. If data is missing, say exactly which variable or pipeline stage is missing.
7. If a fix is needed, propose the safest fix with minimal regression risk.
8. Do not rewrite unrelated code.
9. Preserve working behavior where possible.
10. Prioritize manufacturing correctness over visual prettiness.

## TEST EXECUTION PLAN

Run all of the following checks.

---

# PART 1 — UI BEHAVIOR VALIDATION

Validate the panel behavior in the app UI.

### Check:

* Do buttons appear only after pipeline execution?
* Are buttons hidden before data exists?
* Does selected station button text update correctly?
* Does total station count update correctly?
* Is the header layout stable on small and large screen widths?
* Do buttons overlap title text?
* Are buttons disabled when station data is invalid or empty?
* Does changing selected station update preview and export target together?
* Is there any stale station selection after rerun?

### Must detect:

* hidden state bugs
* stale state bugs
* null data crashes
* wrong selected station export
* header overflow issues

---

# PART 2 — DOWNLOAD FUNCTION VALIDATION

Validate both button actions.

## A. Download Station N

Check:

* Does it download exactly one SVG?
* Is the filename correct?
* Is the selected station number correct in filename and content?
* Does downloaded file open in browser correctly?
* Does repeated clicking cause duplicate or broken files?
* Does it fail silently?

## B. Download All N Stations

Check:

* Does it export all stations?
* Are all stations unique?
* Are filenames unique and ordered?
* Does stagger delay actually prevent browser blocking?
* Is any station skipped?
* Are there race conditions?
* Are wrong station drawings exported due to async closure/state bug?
* Does export break when station count is high?

### Test with:

* 1 station
* 2 stations
* 4 stations
* 8 stations
* 12 stations
* max realistic station count supported by pipeline

---

# PART 3 — SVG FILE STRUCTURE VALIDATION

For every exported SVG, inspect structure.

### Verify:

* valid SVG root
* correct width and height
* proper viewBox
* no clipped geometry
* no off-canvas geometry
* no missing groups
* no broken paths
* no NaN / undefined / null in SVG attributes
* no text overlap in title block
* no dimension arrows outside sheet
* no hidden annotation behind other layers

### Must flag:

* malformed SVG
* invalid numeric values
* rendering only in browser but not in CAD/SVG tools
* missing namespace issues
* print scaling issues

---

# PART 4 — DRAWING LAYOUT VALIDATION

Confirm sheet layout is professional and readable.

### Verify:

* A3 landscape proportions are correct
* border margin is consistent
* double-line DIN frame is correctly placed
* title block fits inside lower/right zone properly
* forming data box placement is stable
* watermark centered and not blocking dimensions
* all three views are placed cleanly
* sufficient white space exists
* no overlapping leaders, arrows, dimension text, or section labels

### Must test:

* short profile
* wide profile
* deep profile
* large OD case
* small OD case

---

# PART 5 — GEOMETRY VALIDATION

This is critical.

You must verify that the geometry shown in each view is consistent with the station tooling data.

## Cross-section nip view

Check:

* upper roll contour matches upper geometry
* lower roll contour matches lower geometry
* strip sits correctly in nip
* strip thickness visually and numerically makes sense
* roll gap is correct
* form depth is correct
* bend angle annotation matches station value
* center alignment is correct
* upper and lower profiles do not unintentionally intersect

## Front view

Check:

* OD circle matches stated OD
* bore size matches title block and dimension callout
* keyway width/depth is correct if represented
* groove depth cut matches tooling values
* concentricity is maintained

## Side view / Section A-A

Check:

* face width matches stated width
* OD matches front view OD
* bore matches front view bore
* groove profile depth matches nip/front values
* section is logically consistent with actual roll body

### Must detect:

* same data labeled differently in different views
* graphic approximation not matching numbers
* incorrect scale between views
* groove depth mismatch
* bore mismatch
* section/profile inconsistency

---
