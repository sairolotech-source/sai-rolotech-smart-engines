==================================================
COPRA-CLASS BENCHMARK TO USE
==================================================

Treat the following as the target benchmark dimensions:

A. Full roll-forming process chain
- calculation
- flower design
- roll tool design
- part lists / BOM
- tool production workflow
- central database / project and tool management

B. Parametric flower design
- editable station count
- station-by-station forming progression
- automatic flower updates after geometry changes
- support for open/closed sections where relevant
- support for repeated shape families / size variants where possible

C. Flower Technology calculations
- technical forming curves
- optimized downhill forming or equivalent strain-aware logic
- 3D wire / centerline / transition model
- engineering calculations, not just visuals

D. Automatic roll design
- SmartRolls-like data-driven roll design from profile cross-section
- minimal manual CAD dependency
- roll contour generation from section logic and process parameters
- station-specific upper/lower roll differences

E. Roll geometry fidelity
- non-hardcoded roll models
- actual contour-driven roll geometry
- if the target roll is non-axisymmetric, geometry must vary with angle as well as axial position
- no fake cylinder fallback unless explicitly labeled fallback
- proper shaft/shoulder/groove logic

F. Simulation / validation
- FEA or simulation module if feasible
- otherwise, implement a simulation-ready validation layer with engineering checks:
  - bend progression consistency
  - strain/elongation approximation
  - interference/collision checks
  - roll gap plausibility
  - forming limit warnings
  - defect heuristics such as twist, corrugation, bow, edge-wave risk
- distinguish clearly between real simulation and heuristic validation

G. Manufacturing outputs
- BOM / part list
- tooling list
- roll dimensions table
- station report
- exportable engineering summary
- downloadable report assets if the app supports file export

H. Database / data model maturity
- central project model
- reusable tooling library / roll library
- profile/project save-load versioning
- traceable input/output data model
- no scattered untyped ad-hoc state for core engineering data

I. CAD / workflow maturity
- DXF / STEP / STL / JSON / CSV export support where realistic
- manufacturing-friendly data structures
- report generation
- reliable import/export boundaries
- error handling, validation, units consistency

J. Engineering credibility
- tests
- deterministic outputs
- no silent fallbacks hiding broken calculations
- no mock geometry presented as final engineering result
- clear limits where physics-grade simulation is absent

SPECIFIC TECHNICAL TASKS TO PERFORM
==================================================

1. Flower engine
Inspect whether the app currently does real flower calculation or just draws shapes.
If only visual:
- implement a real station progression engine
- represent profile progression across stations
- compute station-by-station geometry states
- ensure changes in profile, thickness, width, bend plan, and station count recompute the flower
- support a 3D wire/transition representation if missing
- keep it data-driven and deterministic

2. Forming-curve / downhill / technical calculation layer
If missing:
- add a dedicated engineering module for technical forming calculations
- include at minimum:
  - station progression math
  - bend distribution
  - simple strain/elongation estimation
  - downhill-style vertical progression control or equivalent
  - warnings when inputs exceed plausible limits
Document clearly whether this is:
- true physics simulation
or
- engineering approximation

3. Automatic roll design
Inspect whether rolls are manually modeled, approximated, or automatically derived.
If missing or weak:
- build a roll design generator from profile cross-section and station data
- generate station-specific upper/lower roll contours
- separate design inputs from mesh/view code
- avoid reliance on manual CAD sketches where possible
- add a reusable tooling data structure

4. Roll geometry
Inspect all lower/upper roll generation.
- Remove hardcoded cylinders where still present
- ensure contour data actually drives geometry
- if target shapes are non-axisymmetric, implement geometry as r = f(x, theta), not only lathe/revolve
- otherwise document why axisymmetric geometry is sufficient
- add tests for geometry determinism

5. Validation / simulation-ready layer
If no true FEA exists:
- add an engineering validation module with:
  - station continuity checks
  - roll gap checks
  - collision/interference checks
  - bend severity checks
  - edge strain risk
  - springback placeholder hooks
  - defect heuristics for twist, bow, corrugation, edge wave
- expose results in UI/report
- clearly label as heuristic validation, not FEA

6. Manufacturing outputs
Add or improve:
- BOM / tooling list
- station table
- roll dimensions table
- project summary report
- export to JSON/CSV/PDF or equivalent supported format
- make outputs derived from real engineering state, not mocked display data

7. Central engineering data model
Refactor if needed so the app has a coherent central model for:
- material
- thickness
- profile cross-section
- station count
- station states
- flower data
- roll design data
- tooling data
- validation results
- exports/report data
Add versioning or schema safety if feasible.

8. Persistence / project management
Inspect save/load.
If weak:
- implement robust project serialization
- include engineering inputs, computed outputs, and generated tooling data
- maintain backward compatibility if feasible

9. Testing
Add meaningful automated tests for:
- flower generation
- station progression
- roll contour generation
- export content
- validation warnings
- geometry reproducibility
